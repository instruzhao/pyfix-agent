# v0.6.2 — Scale Validation and Cost Boundaries

v0.6.2 turns repository context from an implementation feature into a measurable benchmark dimension. It adds paired repository-on/off runs, context ground truth, separate repair/review costs, bounded reviewer generation, configurable trace privacy, and nine hint-free multi-module fixtures.

## What changed

- Upgraded the benchmark manifest to schema 3 with non-agent-visible tags, required/relevant context paths, and explicit distractors.
- Expanded the curated benchmark from 15 to 24 cases. The nine new cases cover shipping thresholds, flag layering, inventory boundaries, transitive permissions, half-open bookings, retry backoff, pricing tiers, ordered event identity, and package re-exports.
- Added `--tag` filtering and repeatable `--repository-mode on|off`; specifying both produces paired A/B runs with separate workspace identities.
- Upgraded benchmark reports to schema 4 with repository/legacy success, paired wins/losses/ties, context recall/precision/distractor rate, cache/index timing, budget truncations, and related-file counts.
- Split repair and review input/output tokens, prompt characters, and model durations instead of reporting one blended model cost.
- Added an independent reviewer model instance with a default 3,072-token output cap and 1,024-token thinking budget. Repair generation retains its 8,192/4,096 limits.
- Bounded reviewer JSON to at most three contracts and three risks, with compact evidence and counterexample instructions.
- Added trace privacy modes: `none`, `paths`, and `safe`. The default replaces known local roots; safe mode hashes source-bearing payloads while preserving structured outcomes and metrics.
- Added repository snapshot/build/total timing to trace schema 1.4 and benchmark aggregation.
- Added a generic documented-positive-precondition cue after real qualification exposed a reviewer that silently narrowed an explicit docstring contract.

## Benchmark isolation

Schema 3 context expectations are evaluation metadata only. They are never copied into `BenchmarkCase.agent_task`, repair prompts, review prompts, fixture files, or external holdout execution. They are read only after a run when the runner compares recorded selected paths with benchmark ground truth.

External holdouts remain outside the materialized fixture. Repository-on and repository-off variants each receive a fresh fixture copy and independent Git baseline.

## Compatibility

- Existing schema 1 and 2 manifests remain readable.
- A benchmark without `--repository-mode` uses the configured repository setting exactly once, preserving normal runtime and cost.
- Direct `DefaultAgent` callers can continue sharing the repair model with the reviewer by omitting `review_model`.
- `save_trace(..., redaction_mode="none")` preserves the historical trace payload. The CLI default is `paths`.
- Existing trace fields remain available; schema 1.4 adds redaction and timing metadata.

## Qualification

- All 180 non-integration tests pass. Unit and end-to-end coverage includes A/B identity, schema 3 validation, schema 4 aggregation, cost separation, reviewer bounds, trace redaction, cache timing, and all nine context-ground-truth fixtures.
- `pyfixagent-benchmark --validate` confirms all 24 cases remain isolated failing baselines with external holdouts.
- The final-code `qwen3.6-max-preview` run completed 19 full qualifications with 19 final successes, including all nine new multi-module cases. Five remaining existing cases were blocked when the provider account exhausted its free quota; they are not reported as semantic failures or successes.
- A paired real-model smoke produced one repository win on the package re-export case. The nine-case, four-repetition A/B command remains an optional research protocol rather than a release gate because context-starved legacy retries are materially expensive.
- See `docs/results/v0.6.2-qwen3.6-max-preview.md` for exact costs, retrieval metrics, the false-accept correction, and the provider-blocked cases.
