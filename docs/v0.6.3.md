# v0.6.3 — Provider-Safe Thinking and DeepSeek Default

v0.6.3 separates generic generation limits from provider-specific thinking controls so a model switch does not silently carry incompatible parameters into repair or review calls. The final local release configuration uses the Alibaba Cloud deployment of `deepseek-v4-flash` after the initially selected Kimi endpoint did not return within the bounded smoke-test window.

## What changed

- Changed `model.name` to `deepseek-v4-flash` while retaining the DashScope OpenAI-compatible endpoint and `DASHSCOPE_API_KEY` environment variable.
- Kept thinking mode at temperature `1.0` with standard system-role prompting after a successful real repair call.
- Removed `thinking_budget` from the default repair and semantic-review configuration because that control is provider-specific and must not be inherited implicitly.
- Kept the reviewer output cap at 3,072 tokens and the repair output cap at 8,192 tokens, so both paths remain bounded even without a thinking-budget control.
- Changed reviewer configuration assembly so it never inherits a repair thinking budget. A reviewer budget is sent only when `semantic_review.thinking_budget` explicitly defines one.
- Preserved explicit thinking-budget support for compatible custom configurations such as Qwen deployments.

## Compatibility boundary

The repair engine, semantic reviewer, benchmark runner, and model client still receive the same generic interfaces. Provider-specific choices remain in YAML and CLI composition helpers:

- `enable_thinking` is optional and provider-dependent.
- `thinking_budget` is optional and emitted only when present for repair, or explicitly present in semantic-review configuration for review.
- `system_prompt_as_user` remains available for models that need a merged user-message contract.
- Historical qwen qualification reports remain unchanged and continue to describe the model configuration used for those runs.

## Qualification scope

v0.6.3 is completed and versioned locally. Its offline gate passed all 182 non-integration tests with one real-service integration test deselected. A real `deepseek-v4-flash` demo repair then passed in one iteration, and a separate multi-file re-export case passed visible tests, semantic review, and external holdout in one iteration. These are smoke results, not a full 24-case model qualification. The Kimi attempt produced no model output before the bounded timeout and is treated as an endpoint/availability failure rather than a model-quality result.
