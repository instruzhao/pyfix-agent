## Failure analysis

All five failures passed the visible suite and failed only at the external holdout gate:

- `config_bool` failed once because the repair handled the visible boolean spellings but did not make parsing case-insensitive.
- `file_extension` failed all four repetitions. The repair consistently replaced only the final suffix for the visible multi-dot filename, but did not reliably handle extensionless filenames; it also remained vulnerable to an extension argument that already contained a leading dot.

No case-specific task hint was added to address these failures. During development, a holdout-blind contract-analysis experiment could identify likely counterexamples, but `qwen3.6-flash` did not reliably turn them into correct follow-up edits and the extra workflow increased coupling and token cost. That experiment was not shipped in v0.4.0.

## Interpretation

The 100% visible rate shows that the refactored execution, context, model, backend, retry, trace, and benchmark boundaries preserve the core repair loop. The 91.7% holdout rate also shows why visible pytest success alone is not a sufficient acceptance signal. The remaining gap is semantic generalization, especially for alternate input representations, rather than patch application, workspace policy, or benchmark runner reliability.

This is a small curated local benchmark, not a claim of production-grade repair ability or a comparison with SWE-bench.
