## False-accept finding and correction

An earlier pre-release run exposed a real false accept in `multifile_retry`: visible tests passed, but a positive delay-bound contract was not enforced. The first fixture revision made that contract explicit in the project API docstring rather than adding a case-specific agent hint. A second run proved that the reviewer still silently narrowed the documented contract.

The final implementation adds a generic `declared_positive_precondition` structural cue. It detects a changed public function whose docstring declares positive inputs or bounds while one or more parameters still permit zero or negative values. The cue disappears only after visible non-positive-value guards are present. A boundary-direction regression test also verifies that `x < 0` does not satisfy a positive-only contract. On final code, the retry case produced one blocking review, consumed the validated risk in a semantic revision, changed the guards to `<= 0`, and passed the external holdout with no case-specific prompt.

## Paired A/B smoke and cost boundary

A pre-release paired smoke on `multifile_reexport_tax` used fresh workspaces for repository-off and repository-on variants. Repository-off failed after five invalid edit attempts, while repository-on repaired, passed review, and passed holdout in one iteration, producing one repository win and no losses. The legacy path consumed 22,592 repair tokens; the repository path consumed 3,160 repair tokens plus 2,464 review tokens.

This smoke validates the paired runner and demonstrates a causal context win, but it is not a statistically powered A/B result. A nine-case, four-repetition paired run remains an optional research protocol. It is not a release gate because repeatedly running context-starved legacy variants has high cost and low additional diagnostic value.

## Interpretation

The completed evidence qualifies the v0.6.2 implementation, all nine new repository-context cases, cost separation, retrieval metrics, bounded reviewer generation, trace redaction, and the documented-contract revision path. It does not establish broad repair quality, large-repository scaling, or a complete fresh 24-case real-service result. The remaining five-case gap is explicitly attributable to provider account state and must stay visible until re-run.
