# v0.7.1 — Resource and Supply-Chain Hardening

v0.7.1 closes the highest-priority gaps found during the real v0.7.0 Docker qualification. It changes no model prompt or provider behavior.

## Runtime Policy

- Makes the container backend the default; local execution requires `--sandbox-backend local` and remains restricted to trusted projects.
- Drains container stdout/stderr concurrently into a bounded host buffer and terminates the container when the combined output limit is exceeded.
- Monitors disposable-worktree growth against its initial size and a configurable write budget, including an unthrottled final scan for fast commands.
- Adds kernel-level single-file, open-file, and core-dump limits, private IPC, init-based child reaping, and bounded stop cleanup.
- Records output truncation and policy violations in formatted test evidence.

## Runner Supply Chain

- Keeps the Python base image digest-pinned and makes `65534:65534` the image-level default user. On POSIX hosts, the runtime defaults to the non-root workspace owner's UID/GID so the bind-mounted transaction worktree remains writable; root-owned workspaces are rejected unless an explicit non-root user is configured.
- Separates four reviewed direct requirements into `containers/requirements.in`.
- Locks the complete 20-package Linux/amd64 CPython 3.12 resolution with exact wheel SHA-256 values and installs with `--require-hashes`.
- Adds OCI source/version labels and builds with maximum provenance metadata.
- Verifies that Docker Scout can generate a 156-package SBOM from the final image.

## Vulnerability Policy

Docker Scout currently reports three Critical/High findings inherited from Debian trixie's Perl package. They do not add privileges beyond the already-untrusted non-root workload and have no stable base-image update at qualification time. Each exception has a rationale and an expiry date in `containers/vulnerability-allowlist.json`. `pyfixagent-verify-container` fails on any new finding or expired exception. Public CI uses a commit-pinned Trivy action to reject application-library Critical/High findings without requiring a commercial scanner account, and Dependabot checks the pinned Docker base weekly.

## Runner Selection

The included scientific runner remains intentionally finite. A reviewed project-specific image can be selected with `--container-image` without weakening the runtime `image_only` policy.

## Qualification

The runner image is built and tested locally with Docker Desktop 29.4.2 on Linux/amd64. Docker Scout generated a 156-package SBOM; the expiring CVE gate reports three reviewed base findings, zero unexpected findings, and zero expired exceptions. Real tests cover normal pytest execution, output termination, sampled and final workspace-growth termination, and kernel file-size enforcement.

The complete gate with `RUN_DOCKER_TESTS=1` passes 218 tests with only the opt-in external LLM test skipped. All 24 benchmark fixtures independently validate as isolated failing baselines with external holdouts. A separate real `deepseek-v4-flash` run then repaired the resettable demo in one iteration through the default container backend, with all six visible tests passing. The generated Trace records `success=True`, `workspace_strategy=temporary_git_worktree`, and `environment.execution.backend=container`. The selected demo workspace was restored and independently confirmed at its committed failing baseline of three failures and three passes. The initial integration harness expected the obsolete in-place strategy name; that assertion was corrected without repeating the successful model call.
